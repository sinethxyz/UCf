# Migration Rules

## When Migrations Are Needed

A database migration is required for any of the following changes:
- Creating a new table
- Adding, removing, or altering a column
- Adding or removing an index
- Adding, modifying, or removing a constraint (foreign key, unique, check)
- Creating or modifying a view
- Adding or modifying an enum type

If your change touches any of these, it must go through the migration pipeline. No schema changes via raw SQL or ad-hoc scripts.

## Migration Safety

### Mandatory Reversibility
Every migration must include both `upgrade()` and `downgrade()` functions. A migration that cannot be reversed is rejected. Test both directions before submitting.

### Non-Destructive Defaults
- New columns must have a default value or be nullable. Adding a NOT NULL column without a default is forbidden in a single migration.
- New tables should use sensible defaults for all required fields.
- Index creation should use `CREATE INDEX CONCURRENTLY` when possible to avoid locking.

### Backwards Compatibility During Rolling Deploys
The old version of the application must continue to work while the migration is being applied. This means:
- New columns must be nullable or have defaults (old code won't populate them).
- Removed columns must be dropped only after all application code stops referencing them (two-phase: first remove code references, then drop column in a subsequent migration).
- Renamed columns require a two-phase approach: add new column, migrate data, update code, then drop old column.

## Forbidden Single-Migration Operations

The following operations must NEVER be done in a single migration. They require a multi-step migration plan:

- **Dropping a column referenced by application code** — first remove all code references, deploy, then drop the column in a follow-up migration.
- **Renaming a table** — create the new table, migrate data, update code, then drop the old table across multiple migrations.
- **Changing a column type without data migration** — add a new column with the target type, backfill data with conversion, update code, then drop the old column.
- **Adding NOT NULL without a default** — first add the column as nullable with a default, backfill existing rows, then alter to NOT NULL in a follow-up migration.

## Migration Guard

The migration guard subagent is automatically invoked for any change that touches files under `migrations/`. The guard:
- Validates that both `upgrade()` and `downgrade()` are present and non-empty.
- Checks for forbidden single-migration operations.
- Verifies backwards compatibility with the current application code.
- Confirms the migration can be applied and rolled back cleanly against a test database.

The migration guard verdict is mandatory. A failing guard blocks the PR.

## Task Authorization

Only the following task types are authorized to create or modify files under `migrations/`:
- `migration_plan` — for planned schema changes.
- `canon_update` — for schema changes driven by canon updates.

All other task types are blocked from modifying migration files by hook enforcement. If your task requires a migration but is not one of these types, stop and report the issue. Do not attempt to bypass.

## Migration Naming Convention

Alembic migration files follow this naming pattern:

```
{revision_id}_{short_description}.py
```

- `revision_id` is auto-generated by Alembic.
- `short_description` uses snake_case and describes the change concisely.
- Examples:
  - `a1b2c3d4e5f6_add_company_signals_table.py`
  - `f6e5d4c3b2a1_add_index_on_events_company_id.py`
  - `1a2b3c4d5e6f_add_evidence_strength_column.py`

Generate migrations using `alembic revision --autogenerate -m "short_description"` and always review the generated code before committing.
